# Лабораторна робота №2

## Техніки тест-дизайну, чеклісти, баг-репорти

**Максимальна оцінка:** 15 балів

### Мета роботи

Навчитися застосовувати техніки тест-дизайну, створювати чеклісти для тестування та оформлювати баг-репорти згідно з індустріальними стандартами.

### Предметна область

Ви продовжуєте працювати з системою управління розкладом навчального закладу:
**https://schedule-latest-pz3d.onrender.com**

Використовуйте репозиторій `schedule-testing`, створений у Лабораторній роботі №1.

---

### Підготовка до роботи

1. Додайте нові Labels до репозиторію:
   - `test-design` (синій)
   - `bug` (червоний)
2. Створіть папку `docs/` у репозиторії

---

### Практична частина

#### Завдання 1: Застосування технік тест-дизайну (5 балів)

##### 1.1. Класи еквівалентності та граничні значення (3 бали)

Оберіть поле вводу вашого варіанту та визначте класи еквівалентності і граничні значення.

Створіть GitHub Issue з лейблом `test-design`:

```markdown
Title: [TEST DESIGN] Аналіз поля "Назва поля"

## Опис поля
- Тип: [текст / число / дата / час]
- Обмеження: [мін, макс, формат]

## Класи еквівалентності
| Клас | Опис | Приклад | Очікуваний результат |
|------|------|---------|---------------------|
| Валідний | ... | ... | Успіх |
| Невалідний 1 | ... | ... | Помилка |

## Граничні значення
| Значення | Тип границі | Очікуваний результат |
|----------|-------------|---------------------|
| ... | Нижня -1 | Помилка |
| ... | Нижня | Успіх |
| ... | Верхня | Успіх |
| ... | Верхня +1 | Помилка |
```

##### 1.2. Таблиця рішень (2 бали)

Знайдіть у вашому модулі функціонал, що залежить від комбінації умов, та створіть таблицю рішень.

Створіть GitHub Issue з лейблом `test-design`:

```markdown
Title: [TEST DESIGN] Таблиця рішень: "Назва функціоналу"

## Опис
Що саме залежить від комбінації умов.

## Таблиця рішень
| Умова | Правило 1 | Правило 2 | Правило 3 | Правило 4 |
|-------|-----------|-----------|-----------|-----------|
| Умова 1 | Так | Так | Ні | Ні |
| Умова 2 | Так | Ні | Так | Ні |
| **Результат** | ... | ... | ... | ... |
```

#### Завдання 2: Створення чеклістів (4 бали)

Створіть чеклісти для тестування вашого модуля.

Додайте файл `docs/checklists.md` до репозиторію:

```markdown
# Чеклісти для тестування

## Модуль: [Назва модуля вашого варіанту]

### Функціональність
- [ ] Перевірка 1
- [ ] Перевірка 2
...

### Негативні сценарії
- [ ] Перевірка 1
- [ ] Перевірка 2
...

### UI/UX
- [ ] Перевірка 1
- [ ] Перевірка 2
...
```

**Вимоги:**
- Мінімум 15 перевірок загалом
- Включіть позитивні, негативні сценарії та UI/UX

---

#### Завдання 3: Exploratory testing та баг-репорти (6 балів)

Проведіть сесію exploratory testing вашого модуля та задокументуйте знайдені баги.

**Параметри сесії:**
- Час: 30–45 хвилин
- Фокус: модуль вашого варіанту

**Оформіть знайдені баги як GitHub Issues з лейблом `bug`:**

```markdown
Title: [BUG] Короткий опис проблеми

## Опис
Детальний опис того, що відбувається.

## Кроки відтворення
1. Перший крок
2. Другий крок
3. Третій крок

## Очікуваний результат
Що мало статися.

## Фактичний результат
Що сталося насправді.

## Severity
[Critical / Major / Minor / Trivial]

## Priority
[High / Medium / Low]

## Середовище
- Браузер: ...
- ОС: ...
- Розширення екрану: ...

## Додатково
[Скріншот або відео]
```

**Вимоги:**
- Мінімум 3 баг-репорти
- Хоча б один баг зі скріншотом
- Всі баги додані до Project board

### Варіанти завдань

---

#### Варіант 1: Перегляд розкладу групи

**Модуль:** Відображення розкладу для обраної групи (публічна частина).

**Поле для тест-дизайну:** Вибір групи зі списку (назва групи, навігація по списку).

**Таблиця рішень:** Відображення розкладу залежно від комбінації: обраний тиждень (парний/непарний) × наявність пар у цей день × тип заняття.

**Фокус exploratory testing:** Перегляд розкладу різних груп, переключення тижнів, відображення деталей занять.

---

#### Варіант 2: Перегляд розкладу викладача

**Модуль:** Відображення розкладу для обраного викладача (публічна частина).

**Поле для тест-дизайну:** Пошук/вибір викладача (введення імені, вибір зі списку).

**Таблиця рішень:** Відображення розкладу залежно від комбінації: парний/непарний тиждень × кількість груп у викладача × наявність пар у цей день.

**Фокус exploratory testing:** Пошук викладачів, перегляд розкладу, переключення тижнів, відображення деталей.

---

#### Варіант 3: Додавання та видалення пар у розкладі

**Модуль:** Управління розкладом — розміщення пар на конкретний час/день та їх видалення (адмін-панель).

**Поле для тест-дизайну:** Вибір часового слоту для пари (день, номер пари, тип тижня).

**Таблиця рішень:** Можливість додавання пари залежно від комбінації: слот вільний/зайнятий × аудиторія вільна/зайнята × викладач вільний/зайнятий.

**Фокус exploratory testing:** Додавання пар у різні слоти, спроби створити конфлікти, видалення пар.

---

#### Варіант 4: Управління парами групи

**Модуль:** Налаштування пар для групи — визначення викладача, кількості годин, типу заняття тощо (адмін-панель).

**Поле для тест-дизайну:** Кількість годин для пари (числове поле з обмеженнями).

**Таблиця рішень:** Валідність пари залежно від комбінації: викладач обраний × предмет обраний × тип заняття обраний × кількість годин вказана.

**Фокус exploratory testing:** Створення та редагування пар з різними комбінаціями параметрів, спільні пари з іншими групами.

---

#### Варіант 5: CRUD викладачів та кафедр

**Модуль:** Управління довідником викладачів та кафедр (адмін-панель).

**Поле для тест-дизайну:** Ім'я викладача (текстове поле: довжина, допустимі символи, формат).

**Таблиця рішень:** Можливість видалення викладача залежно від комбінації: має активні пари × має пари в розкладі × прив'язаний до кафедри.

**Фокус exploratory testing:** Створення, редагування та видалення викладачів і кафедр, перевірка зв'язків між ними.

---

#### Варіант 6: CRUD предметів та груп

**Модуль:** Управління довідником предметів та академічних груп (адмін-панель).

**Поле для тест-дизайну:** Назва предмета (текстове поле: довжина, допустимі символи, унікальність).

**Таблиця рішень:** Можливість видалення групи залежно від комбінації: група має пари × група має розклад × група є спільною для іншої пари.

**Фокус exploratory testing:** Створення, редагування та видалення предметів і груп, перевірка валідації та зв'язків.

---

#### Варіант 7: Розподіл занять по аудиторіях

**Модуль:** Перегляд завантаженості аудиторій та виявлення конфліктів (адмін-панель).

**Поле для тест-дизайну:** Номер/назва аудиторії (формат, допустимі символи, унікальність).

**Таблиця рішень:** Відображення статусу аудиторії залежно від комбінації: є заняття в цей час × парний/непарний тиждень × тип аудиторії (лекційна/лабораторна).

**Фокус exploratory testing:** Перегляд розподілу по аудиторіях, пошук конфліктів, навігація по часу та корпусах.

---

#### Варіант 8: Загальна навігація та адаптивність

**Модуль:** Загальний інтерфейс системи, навігація між розділами, адаптивність під різні пристрої.

**Поле для тест-дизайну:** Розмір вікна браузера (ширина у пікселях: мобільний, планшет, десктоп).

**Таблиця рішень:** Відображення меню залежно від комбінації: розмір екрану (мобільний/планшет/десктоп) × орієнтація (портретна/ландшафтна).

**Фокус exploratory testing:** Навігація на різних пристроях та в різних браузерах, зміна розміру вікна, перевірка адаптивності елементів.

### Що має бути в результаті

У вашому GitHub-репозиторії:

```
schedule-testing/
├── docs/
│   └── checklists.md
└── .github/
    └── (GitHub Projects та Issues)
```

**GitHub Issues:**
- Issues з лейблом `test-design` (класи еквівалентності + таблиця рішень)
- Issues з лейблом `bug` (мінімум 3 баг-репорти)
- Всі Issues додані до Project board
