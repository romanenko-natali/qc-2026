# Лабораторна робота №2

## Техніки тест-дизайну, чеклісти, баг-репорти

**Максимальна оцінка:** 15 балів

### Мета роботи

Навчитися застосовувати техніки тест-дизайну, створювати чеклісти для тестування та оформлювати баг-репорти згідно з індустріальними стандартами.

### Предметна область

Ви продовжуєте працювати з системою управління розкладом навчального закладу:
**https://schedule-latest-pz3d.onrender.com**

Використовуйте репозиторій `schedule-testing`, створений у Лабораторній роботі №1.

1. Додайте нові Labels до репозиторію:

2. Створіть папку `docs/` у репозиторії

## Завдання

### Завдання 1: Застосування техніки тест-дизайну (5 балів)

Проаналізуйте функціонал вашого варіанту та оберіть техніку тест-дизайну, яка найкраще підходить:

- **Equivalence Partitioning & Boundary Value Analysis** — для полів вводу з обмеженнями
- **Decision Table** — для функціоналу, де результат залежить від комбінації умов
- **State Transition** — для функціоналу з послідовними кроками та переходами між станами

Створіть **GitHub Issue** з лейблом `test-design`:

```markdown
Title: Аналіз: "Назва функціоналу"
Labels: test-design

## Функціонал
Короткий опис того, що тестуємо.

## Обрана техніка
[EP & BV / Decision Table / State Transition]

## Обґрунтування
Чому саме ця техніка підходить для цього функціоналу.

## Аналіз
(таблиця або діаграма відповідно до обраної техніки)
```

**Вимоги:**
- Обґрунтуйте вибір техніки (чому саме ця, а не інша)
- Аналіз має бути повним (всі класи / всі комбінації / всі стани)

---

### Завдання 2: Створення чеклістів (4 бали)

Створіть чеклісти для тестування вашого модуля.

Додайте файл `docs/checklists.md` до репозиторію:

```markdown
# Чеклісти для тестування

## Модуль: [Назва модуля вашого варіанту]

### Функціональність
- [ ] Перевірка 1
- [ ] Перевірка 2
...

### Негативні сценарії
- [ ] Перевірка 1
- [ ] Перевірка 2
...

### UI/UX
- [ ] Перевірка 1
- [ ] Перевірка 2
...
```

**Вимоги:**
- Мінімум 15 перевірок загалом
- Включіть позитивні, негативні сценарії та UI/UX

---

### Завдання 3: Exploratory testing та баг-репорти (6 балів)

Проведіть сесію exploratory testing вашого модуля та задокументуйте знайдені баги.

**Параметри сесії:**
- Час: 30–45 хвилин
- Фокус: модуль вашого варіанту

**Оформіть знайдені баги як GitHub Issues з лейблом `bug`:**

```markdown
Title: Короткий опис проблеми
Labels: bug

## Опис
Детальний опис того, що відбувається.

## Кроки відтворення
1. Перший крок
2. Другий крок
3. Третій крок

## Очікуваний результат
Що мало статися.

## Фактичний результат
Що сталося насправді.

## Severity
[Critical / Major / Minor / Trivial]

## Priority
[High / Medium / Low]

## Середовище
- Браузер: ...
- ОС: ...
- Розширення екрану: ...

## Додатково
[Скріншот або відео]
```

**Вимоги:**
- Мінімум 3 баг-репорти
- Хоча б один баг зі скріншотом
- Всі баги додані до Project board

## Варіанти завдань

### Варіант 1: Перегляд розкладу групи

**Модуль:** Відображення розкладу для обраної групи (публічна частина).

**Тест-дизайн:** Проаналізуйте, як відображається розклад групи: окреме відображення для непарного та парного тижнів, наявність або відсутність пар у кожному з тижнів, порожні дні.

**Фокус exploratory testing:** Перегляд розкладу різних груп, відображення деталей занять, поведінка коли один із тижнів порожній, коректність інформації в комірках.

---

### Варіант 2: Перегляд розкладу викладача

**Модуль:** Відображення розкладу для обраного викладача (публічна частина).

**Тест-дизайн:** Проаналізуйте, як формується таблиця розкладу викладача: які дні та пари відображаються, як обирається діапазон.

**Фокус exploratory testing:** Пошук та вибір викладачів, перегляд розкладу, коректність відображення для викладачів з різною кількістю пар.

---

### Варіант 3: Додавання пари в розклад

**Модуль:** Розміщення пари на конкретний час/день у розкладі (адмін-панель).

**Тест-дизайн:** Проаналізуйте процес додавання пари: від вибору групи до підтвердження аудиторії. Зверніть увагу на блокування колонок, вибір слоту, вікно аудиторії, поведінку при зайнятій аудиторії.

**Фокус exploratory testing:** Додавання пар у різні слоти, вибір вільних та зайнятих аудиторій, спроби створити конфлікти.

---

### Варіант 4: Управління парами групи

**Модуль:** Створення та редагування пар для групи — предмет, викладач, тип, години тощо (адмін-панель).

**Тест-дизайн:** Проаналізуйте форму створення/редагування пари: поля, обмеження, обов'язковість, залежності між полями.

**Фокус exploratory testing:** Створення та редагування пар з різними комбінаціями параметрів, обов'язкові та необов'язкові поля, групові пари.

---

### Варіант 5: CRUD викладачів та кафедр

**Модуль:** Управління довідником викладачів та кафедр (адмін-панель).

**Тест-дизайн:** Проаналізуйте форму створення/редагування викладача: поля, обмеження, валідація, формат даних.

**Фокус exploratory testing:** Створення, редагування та видалення викладачів і кафедр, перевірка зв'язків між ними, поведінка при спробі видалити викладача з парами.

---

### Варіант 6: CRUD предметів та груп

**Модуль:** Управління довідником предметів та академічних груп (адмін-панель).

**Тест-дизайн:** Проаналізуйте форму створення/редагування предмета або групи: поля, обмеження, валідація, унікальність назви.

**Фокус exploratory testing:** Створення, редагування та видалення предметів і груп, перевірка валідації, поведінка при спробі видалити сутність зі зв'язками.

---

### Варіант 7: Зайнятість аудиторій

**Модуль:** Перегляд завантаженості аудиторій (адмін-панель).

**Тест-дизайн:** Проаналізуйте кольорову індикацію в таблиці аудиторій: за яких умов комірка зелена, жовта чи червона. Зверніть увагу на тултіпи.

**Фокус exploratory testing:** Перегляд таблиці аудиторій, перевірка кольорів при різних ситуаціях, навігація по часу та корпусах, зміна розміру стовпців.

---

### Варіант 8: Створення семестру

**Модуль:** Створення та налаштування семестру (адмін-панель).

**Тест-дизайн:** Проаналізуйте форму створення семестру: поля (рік, назва, дати, дні, пари), обмеження, валідація, чекбокси «Поточний» та «Дефолтний».

**Фокус exploratory testing:** Створення семестрів з різними параметрами, поведінка чекбоксів «Поточний» та «Дефолтний», валідація дат, мінімальні вимоги (дні, пари).

### Що має бути в результаті

У вашому GitHub-репозиторії:

```
schedule-testing/
├── docs/
│   └── checklists.md
└── .github/
    └── (GitHub Projects та Issues)
```

**GitHub Issues:**
- Issue з лейблом `test-design` (обрана техніка + обґрунтування + аналіз)
- Issues з лейблом `bug` (мінімум 3 баг-репорти)
- Всі Issues додані до Project board

### Контрольні питання

1. Яка різниця між класами еквівалентності та граничними значеннями?
2. Коли доцільно використовувати таблицю рішень, а коли — діаграму переходу станів?
3. Чим чекліст відрізняється від тест-кейсу?
4. Що таке exploratory testing і коли його застосовують?
5. Яка різниця між severity та priority?
6. Які обов'язкові елементи має містити баг-репорт?
