# Лабораторна робота №1

## Вимоги та їх аналіз

**Максимальна оцінка:** 10 балів

### Мета роботи

Навчитися аналізувати вимоги до програмного забезпечення, виявляти неповноту та суперечності, формулювати user stories та acceptance criteria.

### Предметна область

Ви працюватимете з  системою управління розкладом навчального закладу:
**http://fmi-schedule.chnu.edu.ua**

Система має два рівні доступу:
**Користувач (публічний доступ):**
- Перегляд розкладу за групами
- Перегляд розкладу за викладачами
- Перегляд розкладу за

**Менеджер (адміністративна панель):**
- Додавання та видалення пар у розкладі
- Управління парами для групи (викладач, кількість годин, тип заняття, групові пари, онлайн-лінк тощо)
- CRUD для викладачів, кафедр, предметів, груп
- Перегляд розподілу занять по аудиторіях

---

### Підготовка до роботи

1. Створіть GitHub-репозиторій `schedule-testing` (або форкніть **https://github.com/romanenko-natali/schedule-for-qc-course**)
2. Увімкніть GitHub Projects для репозиторію:
   - Settings → Features → Projects ✓
3. Створіть новий Project (Board view)
4. Додайте колонки: `Backlog`, `In Progress`, `Done`
5. Налаштуйте Labels в репозиторії:
   - `user-story` (зелений)
   - `question` (жовтий)

---

### Практична частина

#### Завдання 1: **Створення User Stories (5 балів)**

Дослідіть модуль вашого варіанту та створіть **5 user stories** як GitHub Issues.

**Формат Issue:**

```markdown
Title: [USER STORY] Назва функціональності

## User Story
Як [роль],
я хочу [дія],
щоб [цінність].

## Acceptance Criteria
- [ ] Критерій 1
- [ ] Критерій 2
- [ ] Критерій 3
- [ ] Критерій 4
```

**Вимоги:**
- Кожна user story — окремий Issue з лейблом `user-story`
- Мінімум 3–5 acceptance criteria для кожної story
- Criteria мають бути конкретними та тестованими
- Додайте всі Issues до вашого Project board (колонка Backlog)

---

#### Завдання 2: **Аналіз проблем у вимогах (5 балів)**

Проаналізуйте 5 нечітких вимог вашого варіанту. Для кожної створіть GitHub Issue з лейблом `question`.

**Формат Issue:**

```markdown
Title: [QUESTION] Уточнення вимоги: "..."

## Оригінальна вимога
> Текст вимоги

## Проблема
Тип: [неповнота / неоднозначність / суперечливість / нетестовність]

Пояснення чому це проблема.

## Пропозиція
Переформульована вимога, яка є чіткою та тестованою.

## Питання до замовника
1. Питання 1?
2. Питання 2?
```

### Варіанти завдань

---

#### Варіант 1: Перегляд розкладу групи

**Модуль:** Відображення розкладу для обраної групи (публічна частина).

**Область для User Stories:** Вибір групи, відображення пар на тиждень, переключення між парним та непарним тижнем, навігація по тижнях, відображення деталей заняття (предмет, викладач, час, аудиторія, тип).

**Вимоги для аналізу:**

1. Розклад групи повинен відображатися коректно.
2. Користувач може переглядати розклад на будь-який тиждень.
3. Система повинна показувати актуальний розклад.
4. Розклад має добре виглядати на телефоні.
5. Пари в розкладі мають бути зрозумілими.

---

#### Варіант 2: Перегляд розкладу викладача

**Модуль:** Відображення розкладу для обраного викладача (публічна частина).

**Область для User Stories:** Вибір/пошук викладача, відображення пар викладача на тиждень, переключення парний/непарний тиждень, навігація по тижнях, відображення деталей заняття (предмет, група, час, аудиторія, тип).

**Вимоги для аналізу:**

1. Розклад викладача повинен бути доступний усім.
2. Пошук викладача має працювати зручно.
3. Система показує всі пари викладача.
4. Користувач має швидко знаходити потрібного викладача.
5. Розклад викладача повинен бути повним.

---

#### Варіант 3: Додавання та видалення пар у розкладі

**Модуль:** Управління розкладом — розміщення пар на конкретний час/день та їх видалення (адмін-панель).

**Область для User Stories:** Додавання пари для групи на конкретний день/час, вибір типу тижня (парний/непарний/обидва), вибір аудиторії, видалення пари з розкладу, обробка конфліктів (аудиторія зайнята, викладач зайнятий).

**Вимоги для аналізу:**

1. Менеджер може додавати пари до розкладу.
2. Система не повинна допускати конфліктів.
3. Видалення пари має бути простим.
4. Менеджер має бачити, які пари вже додані.
5. Додавання пари повинно працювати швидко.

---

#### Варіант 4: Управління парами групи

**Модуль:** Налаштування пар для групи — визначення викладача, кількості годин, типу заняття тощо (адмін-панель).

**Область для User Stories:** Створення нової пари для групи, прив'язка викладача до пари, вказання типу заняття (лекція/лабораторна/практична), кількість годин, спільні пари з іншою групою, назва на сайті, лінк для онлайн-проведення.

**Вимоги для аналізу:**

1. Менеджер може створювати пари для групи.
2. Пара повинна містити всю необхідну інформацію.
3. Спільні пари мають працювати правильно.
4. Менеджер може редагувати інформацію про пару.
5. Система має контролювати коректність даних.

---

#### Варіант 5: CRUD викладачів та кафедр

**Модуль:** Управління довідником викладачів та кафедр (адмін-панель).

**Область для User Stories:** Додавання/редагування/видалення викладача, прив'язка викладача до кафедри, додавання/редагування/видалення кафедри, перегляд списку викладачів/кафедр, валідація даних.

**Вимоги для аналізу:**

1. Менеджер може керувати списком викладачів.
2. Видалення викладача не повинно зламати розклад.
3. Дані кафедри мають бути повними.
4. Система повинна перевіряти введені дані.
5. Список викладачів має бути зручним для роботи.

---

#### Варіант 6: CRUD предметів та груп

**Модуль:** Управління довідником предметів та академічних груп (адмін-панель).

**Область для User Stories:** Додавання/редагування/видалення предмета, додавання/редагування/видалення групи, прив'язка групи до спеціальності/курсу, перегляд списку предметів/груп, валідація даних.

**Вимоги для аналізу:**

1. Менеджер може додавати нові предмети та групи.
2. Видалення групи має бути безпечним.
3. Предмети мають бути правильно організовані.
4. Система повинна не допускати дублікатів.
5. Групи мають мати зрозумілі назви.

---

#### Варіант 7: Розподіл занять по аудиторіях

**Модуль:** Перегляд завантаженості аудиторій та виявлення конфліктів (адмін-панель).

**Область для User Stories:** Перегляд розкладу аудиторії на тиждень, відображення конфліктів (дві пари в одній аудиторії), пошук вільної аудиторії на конкретний час, фільтрація по корпусу/типу аудиторії, загальна статистика завантаженості.

**Вимоги для аналізу:**

1. Менеджер має бачити розподіл по аудиторіях.
2. Конфлікти аудиторій повинні бути помітними.
3. Система повинна допомагати знаходити вільні аудиторії.
4. Інформація про аудиторії має бути актуальною.
5. Розподіл має бути наочним.

---

#### Варіант 8: Загальна навігація та адаптивність

**Модуль:** Загальний інтерфейс системи, навігація між розділами, адаптивність під різні пристрої.

**Область для User Stories:** Навігація між розкладом груп і викладачів, головна сторінка та структура меню, відображення на мобільних пристроях, відображення на планшетах, коректність роботи в різних браузерах.

**Вимоги для аналізу:**

1. Система повинна працювати на всіх пристроях.
2. Навігація має бути інтуїтивно зрозумілою.
3. Система повинна швидко завантажуватися.
4. Інтерфейс має бути сучасним.
5. Користувач не повинен губитися в системі.
